Gap Analysis
Paper Limitations vs Our Innovation

Original Paper Limitations

No real-time physician control — sensors run continuously with no remote activation/deactivation
Fixed gateway / vendor lock-in — architecture tied to Ubiqsense + Equivital hardware only
No remote sensor activation — patients cannot receive updated sensing profiles from clinicians
Missing alert simulation — no threshold-based warning or notification infrastructure demonstrated
Unidirectional communication only — data flows sensor→cloud, no reverse command channel

Our Innovations

Physician-controlled sensor activation — clinicians toggle individual sensors on/off remotely in real time
Flexible multi-vendor gateway — supports Fitbit, Garmin, or any BLE wearable via abstract gateway layer
Real-time bidirectional communication — WebSocket channel enables cloud-to-device commands
Simulated alert engine — threshold breach triggers instant visual + notification alerts
Modular microservice cloud — Azure p²Health with scalable, replaceable service components

System Design

Technical Architecture

Three-tier IoT architecture enabling seamless data flow from body-worn sensors through a flexible gateway to the Azure p²Health cloud platform.

1
🩺

Sensor Layer

Ubiqsense wristband + Equivital EQ02 chest belt for physiological data. Environmental micro-sensors for gas, noise, UV, and temperature. Fitbit/Garmin as alternate wearable options.

Ubiqsense Equivital Fitbit BLE 5.0
2
📱

Gateway Layer

Smartphone-based gateway aggregates BLE sensor data. Performs local pre-processing, buffering, and AES-128 encryption before forwarding via HTTPS to cloud. Supports multi-vendor wearables.

BLE → HTTPS AES-128 Multi-vendor
3
☁️

Cloud Layer

Azure-hosted p²Health microservices platform. Stores and processes time-series sensor data, provides physician dashboard, alert engine, and bidirectional WebSocket command channel.

Azure OAuth 2.0 WebSocket
